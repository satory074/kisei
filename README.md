# kisei（帰省 — 経路くらべ）

多モーダル（飛行機・新幹線・鉄道・フェリー・バス・車・レンタカー）の経路を実時刻表ベースで列挙し、所要時間と料金を一覧比較する静的 Web アプリ。

**公開 URL**: https://satory074.github.io/kisei/

大阪⇔大間町（下北半島）の帰省のたびに「どの組み合わせが最安/最速か」を手で列挙し直すのが面倒、という課題を解決するために作った。エンジンは汎用（データにある任意の2地点で検索可能）で、初期データは大阪⇔大間コリドーに特化している。

## 使い方

出発地・到着地・出発日時を選んで検索すると、まず**「行き方」の比較**（飛行機（青森）＋レンタカー / 新幹線＋鉄道・バス / 車で直行 …の8とおり前後）が出る。

- 各「行き方」カードに**最短所要・最安目安・行程数**を表示。タップで個別の行程（時刻つき）、さらにタップでレグ別タイムライン（**乗換・待ち時間**を明示）が開く
- **最安 / 最速 / 出発順** でソート。**最速・最安バッジ**は行き方レベルに付く。遠回り・乗継過多の行き方は折りたたみに格下げ
- 行程は「経由: 伊丹空港 → 青森空港」の平文で識別できる。左端が緑の行程は**パレート最適**（より早くかつ安い行程が存在しない）
- **日別料金**: 航空券は日別最安値カレンダー（手動転記）を持ち、選んだ出発日のデータがあれば「**8/12の料金 ¥13,310**」のようにその日の価格で計算・表示する（転記日と30日超の鮮度警告つき）。無い日は従来どおり「目安 ¥X（low〜high）」の幅
- **損益分岐**: 変動運賃（✈️飛行機・🚙レンタカー）を含む行程の詳細末尾に「**航空券を ¥X 以下で取れれば「新幹線＋鉄道・バス」（¥Y）より安くなります**」と平文で表示。予約サイトで実価格がその額を下回るか確認するだけで意思決定できる
- **実価格上書き**: 変動運賃のレグには入力欄があり、予約サイトで見た実際の価格を入れると**総額・順位・損益分岐が即再計算**される（空にすると解除、「実価格をクリア」で全解除）。優先順位は 実価格 ＞ 日別料金 ＞ 幅
- 検索条件は URL クエリに同期されるので、**URL をそのまま共有可能**（例: `?from=osaka&to=oma&time=09:00&sort=cheapest&fares=flight-itm-aoj:18000`。実価格上書きも共有される）

## コマンド

```bash
npm install
npm run dev        # http://localhost:4321/kisei/
npm run build      # 本番ビルド（dist/）
npm run typecheck  # astro check（scripts/ も型検査される）
npm run test       # エンジン+データ検証（smoketest）と DOM テスト（tsx + jsdom）
```

## アーキテクチャ

```
src/
├── data/network.json      # 構造データ（ノード・エッジ・乗換設定・メタ）。単一の真実
├── data/fareCalendar.json # 航空券の日別最安値スナップショット（手動転記、出典・転記日つき）
├── engine/                # 純TS・DOM無し・Dateオブジェクト無し
│   ├── types.ts        # データ/結果の型定義
│   ├── time.ts         # 「出発日0時からの経過分」での時刻演算（"25:30"式の24時超表記対応）
│   ├── validate.ts     # network.json の構造検証（手書き。zod不使用）
│   ├── compile.ts      # 検証→正規化→隣接インデックス構築
│   ├── expand.ts       # timetable/frequency/anytime の「次に乗れる便」計算（日跨ぎはここに集約）
│   ├── search.ts       # DFS + 枝刈り（(ノード×モード構成)別の支配フロンティア）。日別料金も解決
│   ├── pareto.ts       # (到着時刻, typical運賃) の2目的パレート判定
│   ├── group.ts        # 戦略グルーピング: 行程を「行き方」（主要モード列+経由地）単位に束ねる
│   ├── breakeven.ts    # 損益分岐: 変動運賃(flight/rentacar)の許容上限額 + 日本語平文化
│   ├── farecal.ts      # fareCalendar.json の型・検証・参照
│   └── format.ts       # 表示整形の純関数
└── app/                # 唯一のDOM層
    ├── render.ts       # 戦略グループ2段カードの描画 + data-action イベント委譲（open状態復元つき）
    ├── main.ts         # エンジンとUIの配線、URL同期
    ├── fares.ts        # ISO日付演算と fareByDay 構築（エンジンはDate禁止なのでここで解決）
    └── url.ts          # クエリ文字列の encode/decode（純関数）
```

- **探索**: 経路はハードコードせず、エッジ（区間）のグラフを DFS で列挙。エッジごとに「乗れる最初の便」だけを分岐させる（運賃がエッジ単位なので最早便が後続便を支配する）。結果は **パレート集合 ∪ モード構成ごとのベスト**（純パレートだと2〜3件に潰れて「車ならいくら？」の比較ができないため）
- **表示は戦略ファースト**: 検索結果を `group.ts` が「主要モード（飛行機・新幹線・フェリー・車・レンタカー）の列＋経由地」で束ね、第一画面を「行き方の比較」にする。アクセスレグ（鉄道・バス・タクシー・徒歩）の違いでは分裂しない。大阪側の空港（伊丹/関西/神戸）の違いは `cityOf` で同一戦略に統合
- **時刻**: 全便時刻表ベース。のぞみ等の高頻度路線は `frequency`（間隔運転）、車は `anytime` で表現。乗換は「ノード種別ごとの乗換時間 + モード別の乗り込みリード（航空45分・フェリー30分など）」を加算
- **運賃**: 変動運賃（航空券）は `{low, typical, high}` の幅で持ち、typical を主役に幅は脇役表示。加えて日別最安値カレンダー（`fareCalendar.json`）があれば**選んだ出発日の確定価格**で計算する（レグの出発日で引く。日跨ぎ便も出発日の価格）。偽精度を出さない
- **損益分岐と実価格**: データには実価格を取り込まない（価格は「予約時点」に属する情報）。代わりに固定運賃のみの最安の行き方を基準に、変動レッグの許容上限額（損益分岐）を平文で提示する。実価格を入力すると `applyFareOverrides` が当該エッジの運賃を単一値に置換して再探索する（`fares=` クエリで共有可能。データ自体は不変。優先順位: 実価格 ＞ 日別料金 ＞ 幅）

## データの更新方法

`src/data/network.json`（構造）・`src/data/fareCalendar.json`（日別価格）を編集して `npm test`。

- 全エッジに `source`（出典URL）と `lastUpdated`（確認日）が**必須**（validate が強制）
- 概算値・逆算値は `notes` に明記する
- フェリーは季節ダイヤあり（通常ダイヤを収載）。ダイヤ改正期（3月）に見直すこと
- 日別料金はソラハピ最安値カレンダー（`https://www.sorahapi.jp/calendar/{FROM}/{TO}/`）から `byDate` へ転記し、`fetchedAt` を更新（手順の詳細は CLAUDE.md）。転記が古くなっても自動で幅へフォールバックする

## 既知の制限（v1）

- **曜日・季節ダイヤ非対応**（毎日同一ダイヤとして扱う。土日減便・夏期増便は notes 参照）
- **車のフェリー航送ルート非対応**（大阪→青森FT→函館に車ごと乗る経路。車両運賃クラスが複雑なため）
- **1人旅前提**（レンタカー・車の費用は1台あたりのまま合算。`costBasis` はデータに記録済みなので人数対応は将来可能）
- 一部エッジは概算（下北交通バスの運賃、函館のタクシー料金など。notes 参照）
- 青森⇔伊丹の ANA 復路便など、時刻未確認の便は未収載
- 日別料金カレンダーは航空券のみ・転記時点から約4ヶ月先まで（レンタカーは日別定価が無いため幅+実価格入力で運用）
